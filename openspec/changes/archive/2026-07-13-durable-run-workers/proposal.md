# Proposal: durable-run-workers

## Why

The durable-runs subsystem (v0.2, epic #36) is the substrate every message flows through — a message becomes a worker-processed run with a refresh-safe event stream — yet it has **no OpenSpec spec**: it predates the spec-driven flow (built as #48/#50/#107), and its invariants live scattered across `docs/scaling.md`, code comments, and SPEC prose. It's the most load-bearing part of the platform and the least locked-down. Three of its defining behaviors are also still open: the worker processes **one run at a time** (`batchSize: 1`), so a 60-second model stream blocks every other user's run (#117); the worker can only boot **inside the api process**, so it can't scale independently of api replicas (#116); and **run liveness is entirely hand-rolled** — an app-level `setInterval` heartbeat, `runs.heartbeatAt` columns, a `runs.timeouts` queue, and a self-rescheduling liveness poll — reinventing a primitive pg-boss 12 already provides natively (`heartbeatSeconds` + auto-refresh: the monitor fails/retries a job when its worker stops beating). This change locks the durable-runs contract down as a canonical capability spec, adds parallel execution + a dedicated worker entrypoint, and **collapses the bespoke deadman onto the native heartbeat**.

## What Changes

Both capabilities are codified from the shipped code + `docs/scaling.md`, so implementation **verifies (and refactors where reality drifts) against the spec**, not the other way round.

- **New `job-queue` capability spec** — the shared substrate (#47): typed queues (declare/enqueue/consume), **retry + dead-letter** defaults (infra failures retry then dead-letter, never dropped), **admission policy + coalescing** (`stately` + `singletonKey` → one pending + one active per key), **native worker-liveness** (the worker auto-signals liveness while its handler runs; a stalled/dead worker's job is failed and retried, then dead-lettered — no external reaper), and **scheduled (cron) + delayed jobs** — with the guarantee a cron **fires exactly once per tick across all replicas** (single-fire, the reason pg-boss cron beats per-replica `@nestjs/schedule` — sharpened by #116's multiple workers).
- **New `durable-runs` capability spec** — run execution _on_ `job-queue`:
  - message → run → **refresh-safe, replayable event stream** lifecycle (a client is a subscriber to an event log, not a holder of fragile state);
  - **per-chat single-flight** — at most one non-terminal run per chat (`runs_chat_inflight_unique`); a concurrent different message gets **409**, a same-message retry supersedes;
  - **crash/reclaim safety** — `markStarted` refuses terminal runs and reclaims only on a stale heartbeat (CAS), `markFinished` is first-writer-wins;
  - **liveness without a bespoke deadman** — an **in-process wall-clock abort** kills a run that overruns its budget while its worker is alive; **worker death** is detected by the queue's native heartbeat (see `job-queue`) which retries the job so a healthy worker re-executes (the user still gets a result); a run that **exhausts its retries** is settled to a terminal `run.expired` in the owner's tenant scope via the dead-letter path — all with **no cross-tenant reaper**;
  - **fail-fast + self-heal enqueue** — enqueue is not transactional with the run row; a failed enqueue fails the run (freeing the single-flight slot), residual `queued` states self-heal;
  - a **run-level (model) failure** is recorded durably as terminal state and the queue job still succeeds (vs. an infra failure, which `job-queue` retries).
- **New — parallel execution (#117), split across both:** the generic **per-consumer concurrency knob + per-job settlement** (pg-boss native `localConcurrency`; one job's failure MUST NOT settle its siblings) is a **`job-queue`** change; the run **behavior** — runs execute in parallel and per-chat single-flight still holds under concurrency — is a **`durable-runs`** change.
- **New — liveness collapse onto native heartbeat:** delete the hand-rolled deadman — the app-level `setInterval` heartbeat, the `runs.timeouts` queue + its self-rescheduling `checkRunLiveness` poll, and the `runs.heartbeatAt` / stale-threshold columns — replacing them with pg-boss-native `heartbeatSeconds` on the runs queue (worker-death → fail/retry → healthy-worker re-execute), an in-process wall-clock abort for the alive-but-overrunning case, and a `runs.dead` consumer that writes the terminal `run.expired` in the owner's tenant scope on retry-exhaustion. This is a **schema change** (drop the heartbeat columns) and a **run-semantics change** (worker-death now _recovers_ the run rather than immediately expiring it), consciously in scope.
- **New — worker profiles + dedicated entrypoint + independent scaling (#116):** one primitive — a config-declared **worker profile** (a queue-subset + per-queue concurrency), routed by pg-boss's per-queue `work()` (no bespoke router) — powers small installs (one default all-queues process), horizontal fleets (`worker × M` via `docker scale`), and job-class taints (a heavy queue pinned to a capable host). The selective-subscription requirement is **`job-queue`**; the standalone no-HTTP `worker.ts` (composing `QueueModule` + execution modules, one image, graceful `offWork wait:true` drain) and its shared `resolveWorkerProfile()` with the co-located api are **`durable-runs`**. Ship the mechanism + default `all` profile; defer any second named profile to its first real consumer (#196 embeddings).

Not in scope (tracked separately, referenced as roadmap in the design): push-based live deltas (#118) and event-log retention pruning (#119). The spec states the event-stream and retention behavior at the **contract** level; their mechanisms evolve in their own changes.

## Capabilities

### New Capabilities

- `job-queue`: the shared background-job substrate (#47), used by durable runs **and** chat-search alike — typed queues, enqueue/consume with **per-job settlement** and a **per-consumer concurrency** knob (#117's generic primitive), admission policy + coalescing (`stately` + `singletonKey`), bounded retry + dead-letter, and **scheduled (cron) + delayed jobs** — with the load-bearing guarantee that a cron fires **exactly once per tick across all replicas**. pg-boss is the engine (a design/implementation detail, not a requirement).
- `durable-runs`: the worker-processed execution of chat runs **on** `job-queue` — run lifecycle & the refresh-safe event stream, per-chat single-flight, crash/reclaim & deadman liveness, enqueue resilience, parallel per-worker execution (via the queue's concurrency), and the dedicated worker entrypoint scaled independently of the api.

### Modified Capabilities

- _None._ Both `job-queue` (#47) and `durable-runs` (#48/#50/#107) shipped before the spec-driven flow and have no prior spec, so each is codified fresh as a new capability rather than modifying an existing one. `tool-calling`, `chat-search`, `search-projection`, and `instance-config` are unaffected (chat-search/search-projection are **consumers** of `job-queue` — the reindex queue + the sweep cron — which is why the queue is its own capability, not a run-owned one).

## Impact

- `apps/api/src/queue/` — `ConsumeOptions` gains a `concurrency` knob mapped to pg-boss's native `localConcurrency` (per-job settlement, no manual ack — verified in `pg-boss@12.25`); selective queue subscription (already inherent to per-queue `work()`) becomes the routing primitive.
- `apps/api/src/config/` + `apps/api/src/runs/` — a `workers` profile map in `llame.config.json` (profile → `{ queue: concurrency }`, default `all`) + a shared `resolveWorkerProfile()`; `RunsWorkerService` and the search reindex/sweep consumers subscribe only if their queue is in the active profile.
- `apps/api/src/worker.ts` — a new standalone entrypoint (Nest application context, no HTTP) composing `QueueModule` + the run-execution modules, resolving the _same_ profile as `main.ts`; graceful shutdown wiring.
- `apps/api/src/runs/` (liveness collapse) — `QueueOptions` gains `heartbeatSeconds`, set on `RUNS_QUEUE`; delete the `setInterval` heartbeat + `runs.timeouts` queue + `checkRunLiveness`; `executeJob` gains an in-process wall-clock abort; a new `runs.dead` consumer writes terminal `run.expired` in-tenant; `markStarted` reclaim simplifies (native heartbeat, not app CAS).
- `apps/api/src/db/` — a migration dropping `runs.heartbeatAt` (and the now-unused stale-threshold config); the single-flight index, CAS terminal guard, and `run_events` are unchanged.
- `compose.yaml` + operator deployment docs — a `worker` service (`deploy.replicas: N`) alongside `api`, both from one image (`main.js` / `worker.js`), with worker-profile / taint guidance.
- `docs/scaling.md` — constraint 3 (worker concurrency) and the topology table move from "open" to shipped.
- **One DB-schema change** (drop the run heartbeat column) — otherwise concurrency, profiles, and the entrypoint are runtime concerns; the single-flight index and terminal-transition guard already exist. Any drift found between the codified spec and the code is corrected during implementation.
