# Compaction surfacing — show when a long chat was summarized

## Objective

llame compacts long chats (#57): once a chat grows past a threshold, older turns
are folded into a server-generated `summary` (the `compactions` table, with
`upto_seq`) that stands in for those turns in the MODEL's context — the messages
themselves are never deleted, but the model no longer sees them verbatim. This is
COMPLETELY invisible to the user: they have no idea the model's memory of early
turns is now a summary. Surface it — a boundary marker in the message list
("earlier messages summarized for context", with the summary expandable) at the
compaction point. Architecture-to-UX translation (the lever that made
durable-run notifications valuable); read-only and safe. Completes/exposes a
built-but-invisible feature.

## Design

### Backend

- `GET /api/v1/chats/:id/compaction` → the chat's LATEST compaction
  (`CompactionsRepository.findLatestByChatId`, owner-scoped: it JOINs `chats` on
  `owner_user_id` + `compactions_owner` RLS) as `CompactionResponse` (`uptoSeq`,
  `summary`, `createdAt`), or `null` when the chat has none / isn't owned (RLS →
  no row → null; a cross-tenant/absent id is indistinguishable from
  no-compaction — no existence leak). `ParseUUIDPipe` + explicit response type
  (code-first OpenAPI). The latest compaction's `upto_seq` is the current
  boundary — earlier compactions in the lineage are superseded by it.

### Web

- Carry `seq` into the UI message metadata UNCONDITIONALLY (both reviewers'
  P1): `toChatUiMessages` gated `metadata` on `usage` being truthy; restructure
  so `metadata` is ALWAYS present with `seq` (+ `usage` merged when present),
  else user turns (no usage) lose `seq` and get misclassified as post-boundary.
- A `useChatCompactionQuery(chatId, enabled)` — enabled only for a chat with
  messages (a draft never fetches).
- Pure `compactionBoundaryIndex(messages, uptoSeq)` (unit-tested) — the index
  where the marker renders (BEFORE it; `=== messages.length` renders AFTER the
  last). Predicate: the first message where `seq === undefined || seq > uptoSeq`
  (a live/seq-less message IS post-boundary). Sentinel `-1` = no marker, ONLY
  for no-compaction or no-messages. When a compaction exists but EVERY loaded
  message is `seq <= uptoSeq` (all summarized — the most-invisible case), returns
  `messages.length` so the marker still shows (at the bottom). Index `0` is
  valid + common (a reopened long chat's loaded window is all post-boundary →
  marker at the top). Computed over the currently-loaded window (full history
  today — no client pagination yet).
- Render a `CompactionBoundary` at that index (and after the list when it equals
  `messages.length`): a subtle rule + "Earlier messages summarized for context"
  - a disclosure expanding the `summary` as PLAINTEXT (`whitespace-pre-wrap`, NOT
    markdown — so a summary echoing a crafted link/image can't become a beacon).
    Shown whenever a compaction exists (independent of the index value ≥ 0).

## Testability

- API: `GET /chats/:id/compaction` returns the latest compaction for the owner;
  `null` for a chat with none; a cross-tenant id → `null` (owner-scoped, no
  leak); the LATEST (highest `upto_seq`) is returned when several exist.
- Web: `compactionBoundaryIndex` — boundary before the first `seq > uptoSeq`;
  null when no message is past it, when uptoSeq is absent, or when messages lack
  seq; correct index with a mix of seq'd (history) and seq-less (live) messages.
- `toChatUiMessages` carries `seq` into metadata.

## Security invariant (sharing)

The compaction `summary` is generated by feeding the model `partsToText`-rendered
absorbed turns — which `JSON.stringify`s non-text parts, so the summary CAN
contain content the public-share DTO deliberately strips (tool-result fragments,
memory snippets, file paths). Therefore: the compaction endpoint is owner-scoped
(`@ApiBearerAuth`, NOT `@Public`), and the summary MUST NEVER be added to
`SharedChatResponse` / the `/shared` view. There is deliberately no
`compactions_public_read` RLS policy — a public-chat viewer sees messages but
not the compaction boundary (fail-safe / under-permissive), which is correct.

## Non-goals (named)

- Rendering the SUPERSEDED (pre-boundary) messages differently / collapsing them
  (they stay fully visible — compaction is context-only, not deletion; the
  boundary just explains the model's view). Multiple boundaries for a
  multi-compaction lineage (only the latest boundary is shown — earlier ones are
  subsumed). Editing/rewinding a compaction. Showing compaction token cost (the
  usage dashboard's noted follow-up). Live re-render of a NEW compaction
  mid-session (it appears on the next load).
