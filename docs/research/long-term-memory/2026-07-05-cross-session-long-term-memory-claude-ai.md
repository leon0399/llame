# Cross-Session Long-Term Memory for AI Assistants: Verified State of the Art and an Architecture Blueprint for llame

## TL;DR
- **MemPalace is mostly buzz.** It is a real, viral (~47,900-star) open-source project, but its headline "96.6% LongMemEval Recall@5" is a property of plain ChromaDB + `all-MiniLM-L6-v2` embeddings on verbatim text, not its "memory palace" structure; an independent arXiv critique and a code audit found several README claims (lossless 30× compression, contradiction detection, 100% scores) are false or misleading. Do not adopt it. The *durable lesson* it accidentally validates — store verbatim, retrieve semantically, keep the write path cheap — is real and is corroborated by peer-reviewed ablations.
- **The most verifiable, extensible patterns for llame are: (1) a temporal knowledge-graph fact layer with bi-temporal validity/invalidation (Zep/Graphiti), (2) generative-agents-style recency×importance×relevance retrieval scoring with access-timestamp decay, and (3) three-store separation (episodic append-only log / curated semantic / procedural).** All three are peer-reviewed, reproduced, and implementable Postgres-native. The benchmark "wars" (Mem0 vs Zep on LOCOMO) are genuinely unresolved and every single-vendor number should be treated with suspicion — an independent audit found ~6.4% of LOCOMO's answer key is wrong and its judge accepts 62.8% of deliberately wrong answers.
- **Build the decay/provenance hooks NOW, the decay *logic* later.** Every memory record should carry, from day one: `created_at`, `last_accessed_at`, `access_count`, `importance`, `confidence`, `source`/provenance, `owner_scope_type/id`, and bi-temporal `valid_from`/`valid_to`/`invalidated_by`. With those columns present, exponential recency decay, Ebbinghaus-style reinforcement, trust-weighting and contradiction resolution can all be added as pure scoring/query changes without a schema migration.

## Key Findings

### On the specific system you asked about — MemPalace
- **What it is:** A Python, local-first agent-memory tool launched ~April 5, 2026 that organizes memory via the "method of loci" (Wings → Rooms → Closets → Drawers), stores conversation text *verbatim* in ChromaDB, adds a SQLite temporal knowledge graph with validity windows, ships an MCP server, and features a ~170-token "wake-up" cost via a 4-layer (L0–L3) tiered loader. It went viral, reaching over 47,900 GitHub stars and 6,000+ forks within roughly two weeks of its April 5, 2026 launch.
- **Buzz vs. substance:** An arXiv critical analysis ("Spatial Metaphors for LLM Memory," arXiv:2604.21284, Dey et al.) and an independent code audit (lhl/agentic-memory) both conclude: the 96.6% Recall@5 is attributable to verbatim storage + ChromaDB's default embedding model (`all-MiniLM-L6-v2`), *not* the spatial hierarchy (which is ordinary vector metadata filtering, reproducible with a bare ChromaDB collection); the "100% LongMemEval" required undisclosed iterative LLM reranking; the "100% LOCOMO" used top_k=50 (retrieving essentially the whole conversation); "30× compression, zero information loss" (AAAK) is measurably lossy (~84.2% vs 96.6% recall); and "contradiction detection" does not exist in code (the knowledge_graph module implements only exact-match deduplication — conflicting facts accumulate silently). The audit flags MemPalace as "the only system reviewed with multiple outright false claims in the README." The maintainers later self-corrected (v3.3.0), replacing the 100% headline with 98.4% R@5 on held-out data and removing erroneous cross-system comparison tables — to their credit, but the episode is a cautionary tale.
- **Verdict:** Real code, real virality, over-claimed benchmarks. **Not a foundation to build on.** Its one genuinely useful, transferable idea (cheap verbatim write path + semantic retrieval + a small always-loaded "wake-up" context) is better obtained from peer-reviewed sources below.

### The benchmark landscape is unreliable — treat every headline number with suspicion
- **LOCOMO** (Maharana et al., ACL 2024): 10 long multi-session conversations averaging 27.2 sessions each (~16.6K–17.4K tokens per conversation; some sources cite up to ~26K), with 1,540 QA samples broken down as 841 single-hop, 282 multi-hop, 321 temporal, and 96 open-domain (the 5th "adversarial" category is conventionally excluded). Human QA F1 is ~88%. It is the most-cited conversational-memory benchmark and the site of the biggest vendor fights.
- **Independent audit findings (Penfield Labs / dial481 "locomo-audit"):** ~6.4% (99/1,540) of golden answers are wrong, capping the achievable score at ~93.57%; the gpt-4o-mini judge accepts 62.81% of deliberately vague-but-wrong answers; several published systems report scores that mathematically exceed the corrupted ceiling; and a plain full-context GPT-4.1-mini baseline with a good answer prompt (~92.6%) beats the memory systems, implying the answer prompt — not the memory system — often explains the score. The audit also notes category sample sizes range 96–841 (an 8.8× ratio), so most adjacent per-category comparisons are statistically indistinguishable under Wilson-score CIs.
- **The "too short" critique (Continua AI / Letta / Zep):** LOCOMO conversations fit inside modern context windows, so a "no-memory" full-context baseline scores ~73%. LOCOMO largely measures token-efficiency vs. context-dumping, not genuine long-term memory. LongMemEval-S (~115K tokens) has the same problem against 200K–1M context models. Scores are also extremely backbone-sensitive: the same system jumped from 74.4% (gpt-4o-mini) to 84.5% (GPT-4.1-mini) with no other change.
- **LongMemEval** (Wu et al., ICLR 2025, arXiv:2410.10813): 500 questions across five abilities (info extraction, multi-session reasoning, temporal reasoning, knowledge updates, abstention), scalable to ~115K (S) and ~1.5M (M) tokens. Its own paper reports a **30%–60% performance drop on LongMemEval_S** for long-context LLMs; commercial systems (e.g., GPT-4o) reach ~92% on oracle "offline reading" but fall to ~58% (ChatGPT) in full interactive settings — quantifying how badly context length degrades memory. Question mix: 70 single-session-user, 56 single-session-assistant, 30 single-session-preference, 133 multi-session, 78 knowledge-update, 133 temporal-reasoning. It is a *better-designed* benchmark than LOCOMO (explicitly tests knowledge-updates and abstention) and its "indexing → retrieval → reading" decomposition is a good design frame, but it is still context-window-bounded.

### The Mem0 vs Zep dispute — what independent evidence actually shows
- **What happened:** Zep's paper (arXiv:2501.13956) claimed 94.8% DMR (vs MemGPT 93.4%) and strong LongMemEval results, and a Zep blog claimed ~84% on LOCOMO. Mem0's paper (Chhikara et al., arXiv:2504.19413, ECAI 2025) claimed **26% relative improvement in LLM-as-a-Judge over OpenAI memory on LOCOMO (66.9% vs 52.9%), 91% lower p95 latency (1.44s vs 17.12s), and >90% token savings (~1.8K vs ~26K tokens/conversation)** — and reported Zep far lower. Zep replied that its config was broken and a corrected run gave 75.14% (±0.17). Mem0 replied (getzep/zep-papers issue #5) that Zep's 84% wrongly included the excluded adversarial category and used a Zep-favorable prompt, and that a corrected run gave Zep 58.44%.
- **Independent reproductions (matched gpt-4o-mini protocol):** The Hindsight paper (arXiv:2512.12818) reproduces LOCOMO overall as **Memobase 75.78, Zep 75.14, Mem0-Graph 68.44, Mem0 66.88, LangMem 58.10, OpenAI 52.90**. The MemOS cross-system reproduction (widely reused as a neutral baseline) puts MemOS ~75.8, MIRIX 64.33, Mem0 59.22. Letta's own skeptical cross-check put a filesystem MemGPT agent at 74.0 vs Mem0's 68.5 graph variant.
- **The honest read:** Zep/Graphiti and Memobase cluster near the top (~75); Mem0 sits mid-pack (~59–68) in neutral reproductions despite its own much higher self-reported numbers; LangMem and A-Mem are consistently weakest (~40–58). **Mem0's April-2026 "token-efficient" numbers (LongMemEval 93.4%, LOCOMO 91.6%, drifting to 94.4/92.5 by May 2026 on mem0.ai/research) are vendor self-reported on its managed platform, explicitly not reproducible from the open-source SDK, and one neutral floor-reproduction (arXiv:2606.16707, "User as Code") got Mem0 as low as 29–41% under a common backbone.** Do not trust any single vendor's leaderboard.

### System-by-system scorecard (open-source? reproduced? real substance?)

- **MemGPT / Letta** (arXiv:2310.08560; framework open-source, Apache-2.0). *Substance: high.* Foundational OS-inspired "virtual context management" — the LLM self-edits memory via tools across main context / recall storage / archival storage. Now a production framework backed by **Postgres + pgvector with hybrid vector+FTS fused via reciprocal rank fusion** — directly aligned with llame's stack. Adds **sleep-time compute**: background "sleep-time agents" that share memory blocks and consolidate/deduplicate/resolve contradictions asynchronously (documented in Letta's docs and sleep-time-compute blog); Letta Code ships **MemFS, a git-backed memory filesystem** with version history. Reproduced ~74% LOCOMO. *Python-first* (deployment needs Docker/Postgres), so for llame it is a design template, not a drop-in.
- **Zep / Graphiti** (arXiv:2501.13956; Graphiti open-source, powers Zep). *Substance: high.* Bi-temporal temporal knowledge graph: every edge/fact carries `t_valid`/`t_invalid` plus ingestion time; new facts that conflict cause **invalidation, not deletion**, preserving point-in-time queries and auditable episode-level provenance. Three subgraphs (episode / semantic-entity / community) mirror episodic vs semantic memory. Self-hosts on Neo4j/FalkorDB. A community **TypeScript port exists (aexy-io/graphzep)** implementing the paper's architecture, though it is far less battle-tested than the Python Graphiti. This is the strongest published model for the *contradiction-resolution / fact-expiration* piece of your decay roadmap.
- **Mem0 / Mem0g** (arXiv:2504.19413; open-source SDK + managed cloud). *Substance: medium; marketing: heavy.* Clean extract→consolidate→retrieve loop with ADD/UPDATE/DELETE/NOOP operations and an optional graph variant; user_id/agent_id/session scoping; ~1,764 tokens/conversation. Genuinely good latency/token profile and DX. But its SOTA claims are the most disputed in the space and its self-reported numbers are far above neutral reproductions. Python-first; REST API usable from TS.
- **A-Mem** (arXiv:2502.12110; NeurIPS 2025; open-source, ChromaDB). *Substance: medium.* Zettelkasten-inspired: atomic notes with keywords/tags/contextual descriptions, dynamic bidirectional linking, and **memory evolution** (new memories update older linked notes). Conceptually elegant and directly relevant to your "wiki-as-memory + backlinks" idea, but reproduces weakly on LOCOMO (~40s) and adds latency.
- **HippoRAG / HippoRAG 2** (arXiv:2405.14831 NeurIPS'24; arXiv:2502.14802; OSU-NLP, open-source). *Substance: high (peer-reviewed).* Neurobiologically inspired: LLM builds an entity KG (neocortex), Personalized PageRank over it (hippocampal index) enables multi-hop "associativity" and "sense-making." HippoRAG 2 is SOTA on multi-hop QA (MuSiQue, 2Wiki) with open models, cheaper offline indexing than GraphRAG/RAPTOR/LightRAG. Reference implementations run on **PostgreSQL + pgvector + a graph store (Neo4j)**. This is the best-validated *retrieval* algorithm for multi-hop; it is a knowledge-integration RAG method more than a full memory lifecycle.
- **LangMem** (LangChain; open-source SDK, ~746K monthly PyPI downloads). *Substance: medium; ecosystem fit: high for you.* Explicit **semantic / episodic / procedural** memory types with a unified API; hot-path tools + background managers; consolidation to prevent "memory hoarding"; **namespacing by user_id/team_id/app_id** (directly maps to your scopes); pluggable stores incl. Postgres/pgvector. Caveats: **it is Python-only and still pre-1.0 (0.0.30, last released Oct 2025)**, and its LOCOMO latency is pathological (p95 ≈ 59.82s) — a batch/background tool, not an interactive path. Its *conceptual model* (esp. procedural memory as self-editing prompt instructions) is the most useful takeaway.
- **MemOS** (arXiv:2505.22101 / 2507.03724; MemTensor; open-source). *Substance: medium; framing: grandiose.* "Memory OS" with **MemCube** units carrying metadata (timestamps, origin signatures, semantic types), lifecycle governance, permission control, and **multi-cube isolation/controlled sharing across users/projects/agents** — conceptually the closest published match to your shared-vs-private multi-user goal. But its headline "38.98% over OpenAI memory" and 92.34 LOCOMO numbers are vendor-reported; neutral re-runs land ~73–76 (BMAM 73.90, Mnemis 73.3). Useful as a *governance vocabulary*, not a dependency.
- **Cognee** (Markovic et al., 2025; open-source, Apache-2.0, ~17.6K stars). *Substance: medium.* ECL (extract-cognify-load) pipeline building graph+vector memory; API of remember/recall/forget/improve; emphasizes local execution, **tenant isolation, traceability, audit**. Has an OpenClaw plugin. Reasonable option for "company brain"/shared-org memory; Python. (Its "6M memories/month across 100+ companies" is a vendor marketing figure.)
- **Memobase** (open-source; profile-based). *Substance: medium.* Profile/attribute-centric user memory; reproduces surprisingly well (~75.8 LOCOMO, top-tier in the Hindsight reproduction). Good fit for your `profile_memory`/`preference_memory` types.
- **MIRIX** (arXiv:2507.07957; multi-agent, multimodal memory). *Substance: medium.* Multi-agent memory with several memory types; reproduces ~64 LOCOMO. Notable for multimodal/screenshot memory.

### Foundational memory science that is actually validated
- **Generative Agents** (Park et al., UIST 2023, arXiv:2304.03442). The single most reused blueprint: a natural-language **memory stream**; retrieval scored by **recency, importance, and relevance**, combined by summing the three normalized scores with equal weighting (α=1). Recency is specifically **"an exponential decay function over the number of sandbox game hours since the memory was last retrieved," with a decay factor of 0.995**; importance is an **LLM-assigned 1–10 score**; relevance is cosine similarity. Plus **reflection** (periodic synthesis of higher-level insights, triggered when summed importance of recent events crosses a threshold). Every entry stores a creation timestamp and a **most-recent-access timestamp** — the exact hook you need for decay.
- **MemoryBank** (Zhong et al., AAAI 2024, arXiv:2305.10250; open-source). First to operationalize the **Ebbinghaus forgetting curve** with a concrete formula: **Importance = e^(−Δt/S)**, where Δt is time since last retrieval and S is retrieval count (initialized to 1); **on recall, S is incremented by 1 and Δt is reset to 0** — a directly implementable spaced-repetition-like reinforcement-on-access mechanism. Deployed in the SiliconFriend companion chatbot (fine-tuned on ~38k psychological dialogues). Later work (ACM 2026) notes this is a "one-dimensional" simplification (time + recall frequency only), motivating multi-dimensional strength (emotion/novelty/frequency).
- **CoALA** (arXiv:2309.02427) formalizes the working-memory vs long-term (episodic/semantic/procedural) taxonomy and retrieval/consolidation actions — a good conceptual spine.
- **Surveys to anchor the report:** "A Survey on the Memory Mechanism of LLM-based Agents" (Zhang et al., ACM TOIS 2025) is the canonical taxonomy reference; newer 2025–2026 surveys ("Memory in the Age of AI Agents," arXiv:2512.13564; "Rethinking Memory Mechanisms of Foundation Agents," arXiv:2602.06052; "Graph-based Agent Memory," arXiv:2602.05665) extend it and explicitly cover multi-agent shared/private memory.
- **Verbatim vs extracted (peer-reviewed ablation):** "Verbatim Chunks Beat Extracted Artifacts" / CogCanvas (arXiv:2601.00821) is a controlled ablation showing verbatim-grounded storage beats lossy summarization (e.g., 93% vs 19% exact match on a nuanced preference constraint), and that summarization progressively destroys temporal markers. This validates your instinct to keep a raw append-only episodic log and *not* rely on summarization alone.

### Security of agent-written memory — this is a real, measured threat
- **MINJA** (arXiv:2503.03704, NeurIPS 2025): a query-only memory-injection attack — any ordinary user can poison an agent's memory through normal interaction, **achieving a 98.2% average injection success rate** (and >70% end-to-end attack success across GPT-4/GPT-4o agents on EHRAgent/MIMIC-III, RAP/Webshop, MMLU), evading Llama-Guard-style moderation because injected content looks like plausible reasoning. Notably, the authors observe that **temporal decay weakens memory-poisoning attacks** — an additional security argument for building decay.
- **AgentPoison** (NeurIPS 2024, arXiv:2407.12784): backdoor via poisoned records/documents; ≥80% attack success at <0.1% poison rate.
- **MEXTRA / privacy extraction:** stored memories can be leaked through outputs; cross-tenant leakage occurs when memory indexes mix identity domains (arXiv:2606.23075).
- **Defenses that work (arXiv:2601.05504):** two-stage input/output moderation + **trust-aware retrieval with continuous trust scoring at both append and retrieval time** — i.e., a composite trust score per memory. This maps directly onto your scan-on-write policy and your future `support_weight` idea.

### Multi-user shared-vs-private memory — thin but emerging literature (your differentiation opportunity)
- **The core failure mode (named in the literature):** in RAG-backed multi-tenant agents, **retrieval ranks by relevance, not authorization**, so one user's query can surface another user's data (Arceo & Narsing, Red Hat, arXiv:2605.05287). Fix: **enforce isolation at the storage/retrieval layer, not process separation** — mandatory scope filter on every query, per-tenant indexes/namespaces.
- **Dual-tier memory (private + selectively shared) with immutable provenance on every fragment** is proposed in Rezazadeh et al. (arXiv:2505.18279) — essentially your "common vs private vaults" idea.
- **GateMem** (arXiv:2606.18829) is a purpose-built benchmark for **memory governance in multi-principal shared-memory agents** — it argues high recall *without* governance is a security vulnerability, and evaluates access-control enforcement + deletion compliance, exactly your family/team scenario.
- **Multi-agent memory surveys** (arXiv:2602.06052) describe private-per-agent memory + shared workspace with cross-agent read/write rules and conflict resolution.
- **Industry patterns** (AWS Bedrock AgentCore, S3 Vectors; Fastio): hierarchical namespace isolation (global/strategy/tenant/user/session), per-tenant IAM-scoped indexes, mandatory tenant-ID filter on all queries, audit logging of every retrieval, instant per-tenant deletion. LangMem's `user_id/team_id/app_id` namespaces, Zep's user/group graph scoping, and MemOS's multi-cube sharing are the memory-native equivalents.
- **Honest gap:** peer-reviewed, benchmarked shared/private assistant memory is nascent (GateMem is 2026). This is genuinely a differentiation opportunity for llame — but it must be built on hard storage-layer isolation, not LLM discretion.

## Details — recommended architecture for llame

### 1. Validate (with a tweak) the three-layer separation
Your SPEC §20 split — **raw episodic append-only log / curated semantic memory / procedural skills** as separate stores — is well-supported. It matches CoALA's episodic/semantic/procedural taxonomy, Zep's episode/semantic-entity/community subgraphs, LangMem's three memory types, and Letta's recall/archival split. The verbatim-vs-extracted ablation (arXiv:2601.00821) specifically justifies keeping the raw episodic log verbatim and append-only rather than only storing summaries. **Recommendation: keep the three stores, but treat them as three tables/roles in one PostgreSQL, not three databases** — this preserves separation of concerns while honoring your Postgres-first constraint and enabling transactional consistency and joins for permission filtering.

### 2. Postgres-native memory stack (no new infra in MVP)
- **Vectors:** pgvector for semantic memory + episodic-chunk embeddings (Letta and HippoRAG both run on Postgres+pgvector — proof this scales technically).
- **Hybrid retrieval:** combine pgvector similarity with Postgres full-text search, fused via **reciprocal rank fusion** (Letta's exact approach). Add a graph/associativity layer *inside Postgres* using a recursive-CTE or `ltree`/adjacency-table representation of entity links; if multi-hop needs grow, adopt HippoRAG 2's Personalized-PageRank-over-entity-KG pattern (its reference stack is Postgres+pgvector+graph). Qdrant remains your documented scale-swap.
- **Temporal KG facts:** implement Zep/Graphiti's bi-temporal model as a `facts` table with `valid_from`, `valid_to`, `invalidated_by`, `recorded_at`, `provenance_episode_id`. On write of a conflicting fact, **invalidate (set `valid_to`) rather than delete** — this gives you point-in-time queries, auditability, and the belief-revision substrate for later decay.
- **Consolidation/reflection:** run as pg-boss background jobs (your existing queue) — this is exactly Letta's "sleep-time compute" / Supermemory's "dreaming" pattern: a scheduled worker that deduplicates, merges, resolves contradictions, generates reflections/summaries, and updates memory strength. No Redis/cron dependency needed. Letta's practical split is worth copying: a fast model on the interactive path, a stronger/slower model in the background worker.

### 3. TypeScript reality check
- **Implementable in TS today:** the generative-agents scoring formula (trivial), MemoryBank-style decay/reinforcement (the `e^(−Δt/S)` formula is trivial), the three-store schema, bi-temporal facts, hybrid pgvector+FTS+RRF retrieval, scan-on-write defenses, scoped retrieval. The Vercel AI SDK v5 + LangGraph.js handle the agent loop and background extraction.
- **Python-only (use as templates or isolated sidecars, not core deps):** Letta/MemGPT, Mem0 SDK (REST-callable), A-Mem, HippoRAG, Cognee, MemOS, LangMem. **Recommendation: do not take a Python memory framework as a core dependency.** Port the *patterns* (Zep bi-temporal facts, generative-agents scoring, MemoryBank decay, Letta sleep-time consolidation, HippoRAG PPR if needed). Graphiti's TS port (graphzep) can be studied but is not production-grade. If you ever need Mem0/Graphiti's managed features, call them over REST from an isolated worker.

### 4. Decay/extensibility hooks to add NOW (schema-level, logic-later)
Add these columns to **every durable memory record** from day one so decay, trust-weighting, and belief revision are later a *scoring/query change*, never a migration:
- **Temporal:** `created_at`, `updated_at`, `last_accessed_at`, `access_count` (enables exponential recency decay *and* MemoryBank's `e^(−Δt/S)` reinforcement-on-access).
- **Salience:** `importance` (LLM-rated 1–10 at write, per Park et al.), `confidence` (extraction/belief confidence).
- **Provenance/trust:** `source` / `source_type`, `source_trust` (or FK to a source-reliability table), `provenance_episode_id` (link every semantic fact back to the raw episodic event — Graphiti's auditable provenance; also required for your Brain UI "why does it know this?").
- **Bi-temporal validity:** `valid_from`, `valid_to`, `invalidated_by` (contradiction resolution / fact expiration).
- **Scope/permissions:** `owner_scope_type`, `owner_scope_id`, plus a `visibility`/`shared_with` mechanism and permission fingerprint on the search index.
- **Governance:** `memory_type` (profile/preference/project/relationship/task/decision/wiki/ephemeral), `pinned` (never-decay), `forget_after` (TTL for ephemeral_chat_memory).

Your sketched `support_weight = Σ(source_trust × signal_confidence × recency_weight)` is directionally aligned with the literature (it generalizes Park et al.'s recency×importance×relevance and MemoryBank's decay, and adds source-trust which the security literature endorses). But **do not commit to that exact multiplicative form now** — the research shows one-dimensional decay is a known limitation, retrieval quality is dominated by relevance, and naive time-decay can wrongly evict rarely-accessed-but-critical facts (hence `pinned`/`importance` overrides). Ship the *columns*; defer the *formula*; make retrieval scoring a pluggable function (a "boost operator" over the base hybrid search) so you can A/B decay strategies later.

### 5. Wiki-as-memory / Obsidian-style vault
The literature supports a hybrid, not either/or. A-Mem's Zettelkasten notes-with-backlinks and the "LLM-compiled wiki vs vector RAG" comparison (arXiv:2605.18490) show **claim-shaped, human-authored/curated artifacts (wiki pages) retrieve better for synthesis** because they pre-position evidence, while vector chunks align to queries. **Recommendation:** treat Markdown vault files as a *first-class curated-semantic and procedural substrate*, continuously indexed into pgvector + FTS, with backlinks modeled as graph edges in the same entity-link table used for KG facts. Database-backed memory (episodic log, bi-temporal facts, scores) and file-backed memory (wiki/daily notes) coexist: the DB is the index + provenance + scoring layer; the vault is a human-editable projection. This mirrors Letta's git-backed MemFS (versioned file memory) and Cognee's OpenClaw file-first integration. Writing memory to wiki destinations (daily notes, Notion pages) goes through the same approval + scan-on-write gate.

### 6. Security / write-policy (adopt, don't defer)
Implement scan-on-write as trust-aware ingestion (arXiv:2601.05504): dedup, credential/secret screening (never store credentials — hard rule), invisible-Unicode/prompt-injection screening, and a **composite trust score written onto the record** (feeds both retrieval ranking and your future `support_weight`). Configurable thresholds: auto-save / ask / never. Because MINJA operates via normal queries with a ~98% injection rate, also apply **trust-aware retrieval** (don't let low-trust, agent-written memories dominate context) and rely on decay + provenance to contain poisoning. Log every memory read/write with scope IDs for audit (the Brain UI's transparency requirement doubles as your security audit trail).

### 7. Multi-user shared-vs-private (your cherry-on-top)
- **Hard isolation at the storage/retrieval layer:** every retrieval query carries a mandatory scope predicate; deny-overrides-allow RBAC enforced in SQL, not by the LLM. This directly prevents the "relevance beats authorization" leak (arXiv:2605.05287) and the cross-tenant leakage class (arXiv:2606.23075).
- **Scope model:** reuse your `owner_scope_type/id` (user/project/group/instance). "Common" family/team memory = group- or instance-scoped records; private = user-scoped. Retrieval for a given principal = union of (their user scope) ∪ (group scopes they belong to) ∪ (instance scope), filtered by RBAC and permission fingerprint — conceptually MemOS multi-cube sharing / Zep user+group graph / LangMem namespaces, done in Postgres.
- **Governance & deletion:** support per-scope deletion/forgetting and honor it at the retrieval interface (GateMem's behavioral non-recoverability standard). Dual-tier private+shared with immutable provenance (arXiv:2505.18279) is the published design to emulate.
- **Contamination guardrail:** the consolidation/reflection worker must never promote a user-private memory into a shared scope without explicit approval — enforce scope on the *write* side of consolidation too.

## Recommendations (staged)

**Stage 0 — MVP (build now):**
1. Three memory roles in one Postgres: append-only verbatim **episodic** event log; curated **semantic** store (pgvector + FTS); **procedural** skills (prompt-instruction style, LangMem-inspired).
2. Ship the full **provenance/decay/scope column set** (§4) on every record — even though decay logic is off.
3. Hybrid retrieval: pgvector + FTS fused via **RRF**; retrieval scoring as a pluggable function defaulting to relevance-only.
4. **Scan-on-write** trust-aware ingestion (credentials never; dedup; injection/Unicode screening; composite trust score).
5. **Scoped retrieval** with mandatory SQL scope predicate + deny-overrides RBAC; audit-log all reads/writes.
6. **Brain UI** reads provenance (`provenance_episode_id`, `source`, `confidence`, timestamps) so users can inspect/edit/forget.

**Stage 1 — consolidation & temporal facts (next):**
7. pg-boss **sleep-time consolidation** worker: dedup, merge, generate reflections/summaries, resolve contradictions.
8. **Bi-temporal facts** table with invalidation (Zep/Graphiti pattern) for belief revision + point-in-time queries.
9. Turn on **recency×importance×relevance** retrieval scoring (Park et al., γ=0.995 recency decay); flip on **access-timestamp reinforcement** (MemoryBank `e^(−Δt/S)`).

**Stage 2 — decay & shared/private (later):**
10. Introduce configurable decay (exponential recency + Ebbinghaus reinforcement + `pinned`/`importance` overrides); A/B via the pluggable scorer. Only generalize to a `support_weight`-style trust-weighted formula after measuring it on *your* data.
11. Ship **shared vs. private** scopes (group/instance common memory + private vaults) with GateMem-style governance tests.
12. If multi-hop synthesis becomes a bottleneck, add **HippoRAG 2**-style entity-KG + Personalized PageRank inside Postgres.

**Benchmarks/thresholds that would change these choices:**
- Build your **own** eval set from real llame transcripts (LOCOMO/LongMemEval are context-window-bounded and, per the audit, partly corrupted — use them only as sanity checks, never as the target).
- If a "no-memory" long-context baseline matches your memory system on your eval, the memory system isn't earning its complexity — simplify.
- If consolidation latency or cost dominates, move heavier extraction to the sleep-time worker (fast model interactive, strong model in background).
- Adopt Qdrant only when pgvector recall/latency degrades at your data scale.

## Caveats
- **Every headline benchmark number in this space is suspect.** LOCOMO has a ~6.4%-erroneous answer key, a lenient judge (accepts 62.8% of wrong answers), and conversations short enough to fit in context; vendor numbers (Mem0, MemOS, MemPalace) are self-reported, drift upward across blog revisions, and are highly sensitive to backbone model, judge, answer-prompt verbosity, and vector store. Independent reproductions differ from vendor claims by 20–60 points in the worst cases (e.g., Mem0 92.5 self-reported vs 29–41 in a neutral floor reproduction).
- **"Independent" is relative.** The most-cited cross-system LOCOMO reproduction ("Li et al.") originates from the MemOS team, not a fully disinterested third party; the cleanest neutral reproductions relied on here are the Hindsight paper (arXiv:2512.12818), the locomo-audit (Penfield Labs/dial481), and Continua's "LoCoMo Fair Fight."
- **Vendor blog claims flagged as unverified:** Mem0's 26%/91%/90% headline and its 93.4/91.6 (later 94.4/92.5) token-efficient numbers; MemOS's 38.98%-over-OpenAI and 92.34 LOCOMO; MemPalace's 96.6%/100%/30×; Cognee's "6M memories/month." Treat all as marketing until independently reproduced.
- **Shared/private multi-user memory is under-researched** — GateMem (2026) is the first governance benchmark; expect to be partly building on first principles.
- **Dates:** MemPalace, the Mem0 token-efficient update, and several 2026 arXiv items are very recent, and some (e.g., arXiv:2604.21284, 2606.16707, 2601.00821, 2606.18829) are non-peer-reviewed preprints; treat preprint numbers as provisional. Peer-reviewed anchors you can rely on most heavily: MemGPT (arXiv:2310.08560), Generative Agents (UIST 2023), MemoryBank (AAAI 2024), HippoRAG (NeurIPS 2024), A-Mem (NeurIPS 2025), LongMemEval (ICLR 2025), Mem0 (ECAI 2025), the Zhang et al. memory survey (ACM TOIS 2025), and MINJA (NeurIPS 2025).
