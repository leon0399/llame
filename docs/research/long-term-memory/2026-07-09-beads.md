That framing means *working memory for long-horizon tasks* — an issue graph — not episodic/semantic memory in our sense. ## 1. Dependency model with readiness semantics

Four dependency types, split by whether they gate readiness (`docs/ARCHITECTURE.md:196-202`):

| type | readiness effect | semantics |
|---|---|---|
| `blocks` | gates | X must close before Y is ready |
| `parent-child` | gates | epic/subtask; children blocked while parent is blocked |
| `related` | none | soft cross-reference |
| `discovered-from` | none | created while working on the parent — **first-class provenance for agent-generated tasks** |

Status vocabulary: `open, in_progress, blocked, deferred, closed, tombstone, pinned, hooked`; soft-delete via tombstone fields (`deleted_at`, `deleted_by`, `delete_reason`) for synced issues — invalidate-don't-delete, same instinct as our bi-temporal facts.

**Ready-work is a deterministic SQL query, not app-level graph traversal** (`internal/storage/issueops/ready_work.go:93-140`): a built `WHERE` clause excluding issues blocked by open gating dependencies, with precomputed ID sets for deferred-parent children and — notably — **transitive** parent descendants. The comment preserves the bug story (GH#3396): parent filtering was originally a one-hop subquery, silently dropping grandchildren while the docs promised "descendants (recursive)". Lesson for llame's todos: if the API says subtree, the predicate needs a recursive CTE from day one; one-hop is a latent lie. Advertised at ~10ms offline — validates computing readiness in Postgres directly, no dependency-resolution service.

Distributed ID allocation: content-hash-derived short IDs (4→6 chars, progressively lengthened) instead of sequential IDs, so two clones can't both mint `bd-10`; merge = same id+hash → skip, same id+different hash → newer wins, else create. llame is centralized Postgres — not needed — but it's the correct shape if offline/multi-instance sync ever lands.

## 2. Lease / heartbeat / reclaim (agent run ownership)

`internal/storage/issueops/lease.go` — the pattern our pg-boss durable runs can borrow wholesale:

- `ClaimReadyIssue` atomically claims (assignee + `lease_expires_at = now + TTL`); `DefaultLeaseTTL = 5min`, overridable per claim.
- Workers heartbeat well within the TTL; a dead worker stops heartbeating, the lease goes stale, and `bd reclaim` reverts the issue to ready. No coordinator process — liveness is inferred from the data.
- **The `row_lock` keystone** — the subtle part. Dolt has no row locking and merges concurrent commits *cell-by-cell*, so a heartbeat (writing `heartbeat_at`) and a close (writing `status`) on the same row would merge silently — letting a reclaim quietly revert an issue its owner had just closed. Fix: every status/ownership/lease-mutating path rewrites one shared `row_lock` cell to a fresh crypto-random value, forcing those writers to collide and trigger the serialization-conflict retry path. The file states the invariant explicitly: *any* path mutating status/assignee/lease columns MUST rewrite `row_lock`; orthogonal-cell paths intentionally don't.

Postgres gives llame real row locks, so the `row_lock` trick itself is unnecessary — but the transferable discipline is the **explicitly documented co-mutation invariant**: enumerate which column families race against reclaim/heartbeat, and make every mutator of that family serialize on the same conflict point. Our run-state transitions (`runs.status`, worker ownership, lease columns if we add them) deserve the same explicitness, e.g. a single `FOR UPDATE` target or version column named in a schema comment.

## 3. Wisp → digest compaction

`docs/ARCHITECTURE.md:294-334`, `docs/MOLECULES.md` — **molecules** are template workflows; spawning one creates **wisps**: ephemeral child issues tracking execution steps that live *only in the spawning agent's local DB*. They are never exported or synced ("they cannot resurrect from other clones — they were never there"), and `bd mol squash` hard-deletes them (no tombstones needed) after compressing the trace into **one permanent digest issue**.

This is the cleanest shipped form of the trace-compaction idea: cheap noisy execution trace, private to the executor, collapsed to a durable summary at completion, raw steps discarded by design. It rhymes with baro's run-decision-document lesson and maps directly onto llame's run event streams: the replayable event log is the wisp-equivalent (we keep ours for replay/audit), and a `run_digest` — produced at run completion, feeding `memory.consolidate` as an extraction source — is the squash. Also note the elegant deletion-correctness argument: never-synced data needs no tombstones; deletion semantics are a function of replication scope. (Our per-user non-shared rows enjoy the same simplification; shared-scope rows don't.)

Complementing this, `bd prune` is **reference-aware**: closed issues still cited in open issues' text/comments are skipped by default. Good invariant for our `memory_citations` table — a cited memory (or cited episodic range) is not eligible for hard-delete/prune while citations stand.

## 4. Anti-examples (memory chapter)

1. **`bd remember` + `bd prime`: the dump-everything injection.** The memory feature is a flat KV store (`kvPrefix + key → content`, key auto-slugified, update-in-place, substring search). `bd prime` — designed to run as a SessionStart hook — renders **all** memories into a `## Persistent Memories` markdown section: raw prose under key headers, **no** recalled-data framing, **no** provenance/citations, **no** confidence or lifecycle, and — because the store keeps **no timestamps** — entries are ordered *alphabetically by key*. No recency, no relevance, no usage signal; the third dump-all specimen after baro and the pattern our threshold-gated injection exists to avoid.
2. **No access control anywhere.** Identity is a free-text `actor`/`assignee` string plus a git-author string on Dolt commits; no authentication, no RBAC, no ACL — anyone who can reach the Dolt DB can read/write/delete everything. Coherent for single-repo dev tooling; the exact opposite of llame's tenancy model, and disqualifying as a multi-user reference.

## 5. Micro-patterns worth keeping despite §4

Two details inside the otherwise-anti-example `prime` renderer (`cmd/bd/prime.go:430-500`) are genuinely good:

- **The elision banner.** Caps (`--max-memories`, `--max-memory-chars`) apply at whole-memory boundaries, at least one entry is always emitted, and when entries are elided a banner states *how many were dropped, which cap fired, and the exact command to browse the rest* ("N more not shown; browse with `bd memories <keyword>`"). A capped injection never silently pretends to be complete. Our eager-injection budget should do the same: when the token budget truncates recall, say so in the injected block and point at the `memory_search` tool.
- **Injection carries its own usage instructions** ("Stored via `bd remember`. Update in place with… Search with… Remove with…") — the tools demo riding along with the injected memory, which is precisely the one-shot-example pattern from our §7 addendum, independently arrived at.

## 6. Relevance to llame

Beads is a strong reference for the **goals/todos/runs** entity model and a contrast case for memory. Adopt into the todos/runs design when those specs are written: the four-type dependency vocabulary with readiness semantics (especially `discovered-from` provenance), ready-work as a recursive-CTE-backed SQL predicate, lease+heartbeat+reclaim for durable run ownership with a documented co-mutation invariant, wisp→digest compaction as the template for run-event-stream summarization, and reference-aware pruning. Into the memory design: the elision banner, injection-carried tool instructions, and the two anti-examples above.
