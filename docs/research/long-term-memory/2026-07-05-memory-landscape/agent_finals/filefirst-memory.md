# Agent: filefirst-memory — File-first / Obsidian-style memory (delivered 2026-07-05T13:56Z)

## (a) Synthesis

A well-executed "brain/data-first" memory design treats plain files (usually Markdown + YAML frontmatter) as the *source of truth* and everything else — SQLite FTS5, vector indexes, in-memory graphs — as a *derived, rebuildable cache*. The invariant that makes this work: delete the index, rebuild it from the files, lose nothing. This gets you three properties a DB-native design doesn't: (1) human/git legibility — a person or another tool can read, diff, and edit memory without a client; (2) crash-safety by construction — the cache can be stale or missing without corrupting truth; (3) natural sharding — one file per fact/topic means partial writes never corrupt unrelated memories, and lazy-loading (read the index, `Read` individual files on demand) keeps context cost bounded regardless of total memory size.

OpenClaw is the cleanest real implementation of this pattern seen. A canonical `MEMORY.md` seeds each workspace root (`src/memory/root-memory-files.ts`, symlink-rejecting, legacy-path-aware). The actual index isn't built in-process — OpenClaw shells out to an external `qmd` CLI binary (`packages/memory-host-sdk/src/host/qmd-process.ts`, spawn-based, with timeout/kill handling) which owns a SQLite DB via Node's built-in `node:sqlite` plus the `sqlite-vec` loadable extension for vectors (`engine-storage.ts` → `host/sqlite-vec.ts`). When no embedding provider is configured it degrades gracefully to FTS-only with keyword-extraction query expansion tuned for conversational queries (`host/query-expansion.ts`) rather than failing. Write triggers are context-pressure-driven, not scheduled: `memory-flush.ts` builds flush/compaction prompts when the live transcript would exceed the model's token budget — consolidation is reactive to token economics, not a cron job. A scope-rule system (`qmd-scope.ts`) gates which channels/chat-types are even eligible for indexing — an allowlist model, useful precedent for multi-tenant gating.

Hermes Agent is architecturally different from what was assumed: `memory_manager.py` is a *provider-orchestration facade* (one built-in + at most one external plugin, e.g. the third-party Hindsight/OpenViking services) — not itself a 3-layer episodic/semantic/procedural store. Hindsight's own taxonomy (raw `world`/`experience` facts vs consolidated, deduplicated `observation` layer) is the closest match to episodic→semantic promotion, but that logic lives in the external service, not Hermes's code. What Hermes does own and do well: recall-time-only sanitization (`sanitize_context`, `StreamingContextScrubber`) that strips injected framing from recalled memory and wraps it in an explicit "NOT new user input" system note — a prompt-injection defense applied at read time, with no write-time content scanning; and session lineage via a `parent_session_id` FK chain in SQLite (`hermes_state.py`), used for compression-triggered session splitting.

Claude Code's auto-memory (undocumented at launch, v2.1.32) is the "one fact, one file" pattern in its purest form: `~/.claude/projects/<project>/memory/MEMORY.md` index plus per-topic files with SKILL.md-style frontmatter (`name`/`description`/`type`/`originSessionId`), index loaded every session, individual files lazy-`Read`. It has real, documented limits: MEMORY.md is truncated past 200 lines/25KB, and only 5 memories load per query regardless of corpus size — no hybrid vector+keyword fusion at all, purely description-string matching. This is consistent with Anthropic's broader agentic-search bet: the well-corroborated claim that Claude Code dropped an embeddings/vector-DB RAG pipeline for grep/glob/file-read and it "outperformed everything, by a lot" (Boris Cherny), later partially validated academically (Amazon Science AAAI'26: agentic keyword search ~94.5% of RAG faithfulness, zero vector store). Counter-evidence exists too: harness design moves accuracy more than retrieval algorithm choice in controlled tests, and DeepMind has a theoretical result that fixed-dimension embeddings have a hard cap on representable query-document relations — arguments for keeping vectors as a *complement*, not proof grep suffices at unbounded scale.

Trade-off for llame specifically: file-first is the right model for **Knowledge Spaces / artifacts** (already your design — vaults, notes, git repos as knowledge substrate) where content is naturally document-shaped and per-user/project scoped. It is the *wrong* model for **chat/turn memory in a multi-tenant, RLS-enforced platform** — files on a shared filesystem have no per-request tenant boundary, no transactional consistency with the rest of the schema, and no policy-based access control equivalent to Postgres RLS. The pattern worth borrowing isn't "replace Postgres with files," it's "treat structured memory rows as the source of truth and add a derived FTS/vector cache that can be dropped and rebuilt" — which is just applying the same file-first invariant to your existing DB-owned model (a `memory_facts` table + tsvector/pgvector index that's provably reconstructable from the table, rather than a bolted-on external vector service). Sleep-time/consolidation (Letta) and episodic→semantic promotion (RecMem, SCM) are patterns that transfer regardless of storage substrate — decouple online logging (fast writes) from a background job that clusters, summarizes, and promotes to durable structured facts.
