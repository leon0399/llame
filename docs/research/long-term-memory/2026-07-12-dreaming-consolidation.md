# Dreaming / sleep-time memory consolidation — landscape + verdict (2026-07-12)

**Status:** Reviewed; verdict adopted (no consolidation layer for llame now — rationale in §7)
**Method:** two parallel research agents — a code-level dive into OpenClaw's shipped `dreaming` module (checkout at HEAD `f7e0a61`, 2026-07-12) and a web/literature landscape sweep — synthesized and cross-checked against the [2026-07-05 memory-landscape cross-report](./2026-07-05-memory-landscape/CROSS-REPORT.md) and the episodic-memory track ([`docs/research/chat-search/`](../chat-search/2026-07-12-chat-search-cross-report.md), tracker #194).
**Question under test:** does offline/idle "dreaming" consolidation (re-processing conversation history into distilled durable memory) earn a place in llame — and if so, on which side of the episodic (§#194) / curated (SPEC §20) split?

---

## 1. Disambiguation first: three different things share the name

The single most useful output of this research is that "dreaming"/"sleep-time" is **three unrelated mechanisms** wearing one brand. Conflating them (as the popular press does) produces wrong conclusions:

1. **Speculative pre-computation** — Berkeley/Letta "Sleep-time Compute" ([arXiv:2504.13171](https://arxiv.org/abs/2504.13171)): during idle time, pre-compute reasoning about _predictable future queries_ so the live turn is cheaper. Real evidence (~5× test-time compute reduction, up to +13% accuracy on Stateful GSM-Symbolic/AIME) — but the paper's own load-bearing caveat is that it works **only when the future query is predictable from existing context**. This is response pre-computation, not memory consolidation. Orthogonal to the memory question.
2. **LLM re-writing of stored memory** — reflection trees (Generative Agents, [arXiv:2304.03442](https://arxiv.org/abs/2304.03442)), NREM/REM-styled batch rewriters (SCM), "active dreaming" (ADM). This is what "dreaming" intuitively suggests. §3: the evidence for it is weak-to-absent on shared benchmarks.
3. **Deterministic promotion of already-written verbatim snippets** — what OpenClaw's shipped `dreaming` actually is (§4) and, structurally, gbrain's cycle triage (§5). No LLM judges memory content; arithmetic over retrieval telemetry decides what gets durable. The name is marketing; the mechanism is a ranking job.

**Pattern:** no mainstream production memory system runs genuine LLM-batch consolidation; Zep and current Mem0 have moved in the opposite direction (provenance-preserving raw/structured storage). The only shipped "dreaming" implementations we could verify in code are mostly-deterministic promotion/maintenance pipelines.

## 3. Evidence: does offline extraction beat verbatim? (No.)

- **The key ablation** — "Diagnosing Retrieval vs. Utilization Bottlenecks in LLM Agent Memory" ([arXiv:2603.02473](https://arxiv.org/html/2603.02473v2)), LoCoMo, write strategy isolated from retrieval strategy: **write strategy explains 3–8 accuracy points; retrieval method explains ~20** (57.1→77.2). Raw chunked storage (zero LLM calls) at 77.9/81.1 (cosine/hybrid) **beats** Mem0-style extraction (72.2/77.3) and MemGPT-style summarization (70.1/73.3) at matched retrieval; retrieval precision correlates with accuracy at r=0.98. This strengthens the 2026-07-05 cross-report's verbatim-beats-extraction verdict (arXiv:2601.00821) with a second, independent, better-controlled result.
- **The 2025–26 dreaming papers don't rebut it**: SCM ("sleep-consolidated memory", NREM/REM FSM) reports "perfect recall" on an 8-test in-house suite up to 360 concepts with a self-admitted bug that had silently disabled forgetting — a research preview, not evidence. ADM beats Reflexion/MemGPT with large effect sizes but bundles 3–5 LLM calls per consolidation cycle with unquantified cost. RecMem ([arXiv:2605.16045](https://arxiv.org/pdf/2605.16045)) argues for batching consolidation to cut **cost**, not to improve quality. None reports beating the raw-verbatim baseline on a shared external benchmark. That absence is the load-bearing fact.
- **Scale caveat (honest unknown):** these benchmarks test tens of sessions / ~100k-token histories. Nothing tests years-of-history scale, where pure verbatim retrieval may eventually degrade (MemBench signals). "Verbatim wins" is established at benchmark scale, open beyond it.
- **Vendor claims to quarantine:** OpenAI "Dreaming V3" (claimed June 2026: idle-window consolidation, temporal self-updating, 9.4→75.1% time-sensitive accuracy on an _internal_ eval) — primary source unreachable during this research; all numbers second-hand from blogs that may be citing each other. Directional signal at most. **"Claude Code AutoDream" is very likely not a real feature** — every source is a third-party blog citing other third-party blogs with suspiciously specific unverifiable internals; treat as a content-farm citation loop and never cite. The meme's existence is itself a data point: "dreaming" is now a marketing frame detached from shipped mechanisms.

## 4. OpenClaw's `dreaming` — code-verified: deterministic promotion, cosmetic LLM

Module: `extensions/memory-core/src/{dreaming.ts, dreaming-phases.ts, short-term-promotion.ts, dreaming-narrative.ts, …}`; doc `docs/concepts/dreaming.md`. Shipped, heavily tested (~12.5k test lines vs ~14.2k source lines, 88 changelog entries, a dozen fixed production incidents), **opt-in and disabled by default**.

- **Trigger:** managed cron (`0 3 * * *` default), reconciled every 60 s; fires a silent system-event turn into an isolated session (`dreaming.ts:165-189`, hook gate at `:947-984`). No idle detection.
- **The pipeline is arithmetic, not LLM.** Every `memory_search` tool call records which snippets were **surfaced** into a short-term recall store (`tools.ts:679-687` → `recordShortTermRecalls`). A daily sweep scores snippets on frequency/relevance/diversity/recency, applies an **AND-gate** (`minScore` ∧ `minRecallCount` ∧ `minUniqueQueries`, `short-term-promotion.ts:1878-1918`), re-validates against the _latest_ store state at apply time (closing a TOCTOU window, `:2453-2469`), rehydrates candidates from live files (dropping deleted/edited-away text), and **appends** winners to `MEMORY.md` with a `promotedAt` idempotency marker plus an audit event (`:2422-2570`). Oldest-section budget compaction is the only deletion, and it's blunt char-budget pruning, not relevance-aware forgetting.
- **The only LLM call writes a diary.** `dreaming-narrative.ts:65-89,281-325` — a whimsical first-person dream-diary entry (80–180 words) for a UI tab, explicitly non-authoritative and excluded from promotion eligibility (`isContaminatedDreamingSnippet`, `short-term-promotion.ts:417-445`). There is no insight-extraction, dedup, forgetting, or reflection prompt anywhere; "REM reflection" is deterministic string/tag clustering. Shadow trials (`dreaming-shadow-trial.ts`) are report-only scaffolding with **zero call sites** in the production scoring path — the docs' claim that trials "layer on top of the base score" has no corresponding code.
- **Recall path is flat:** promoted `MEMORY.md` re-enters context through the **same** `memory_search` corpus as raw daily notes and transcripts (`prompt-section.ts`) — no wholesale injection, no separate curated channel. Promotion buys durability + a naming in the search-first instruction + a small stickiness boost, nothing else.
- **Honest gaps:** the "recall" signal is _search-surfaced_, not _actually-used_ (an impressions proxy, not usage proof); redaction is secrets-regex only (docs overstate it as "personal and sensitive content"); the contamination guard prevents self-feedback loops but is **not** an injection defense — a malicious transcript fact that keeps getting surfaced can still be promoted (the `minUniqueQueries`/multi-day diversity gate only partially mitigates).

**Reframe this forces:** OpenClaw's dreaming is neither verbatim-retrieval nor LLM-extraction — it's a third category, **usefulness-weighted promotion of already-written verbatim snippets**, where the entire memory-judgment layer is free (plain code) and the LLM is decoration. That sidesteps, by construction, every consolidation failure mode in the literature (poisoning laundering via rewrite, paraphrase drift, compounding hallucination, silent LLM spend).

## 5. gbrain's `dream` cycle — correction to the 2026-07-09 deep dive

The [gbrain deep dive](./2026-07-09-gbrain.md) covered the synchronous facts path and missed that gbrain **also ships a scheduled offline cycle** (verified first-hand 2026-07-12; addendum added to that doc): `gbrain dream` (`src/commands/dream.ts` — "cron: `0 2 * * *`"; thin alias over `runCycle`, `src/core/cycle.ts`) runs a multi-phase maintenance pipeline (lint → backlinks → sync → **synthesize** → extract → patterns → emotional-weight recompute → embed → orphans report). The dreaming-relevant phase is **synthesize** (`src/core/cycle/synthesize.ts`): discovered conversation transcripts get a cached cheap-model "worth processing?" verdict (Haiku, cached in a `dream_verdicts` table), and only worth-processing transcripts fan out to a stronger model (Sonnet subagent) that distills pages into the brain. So gbrain = **deterministic maintenance + LLM triage gate + LLM distillation for the minority that passes** — a hybrid between OpenClaw's pure arithmetic and the literature's full-LLM consolidation, with the cost-control lesson (cheap-model gate, verdict cache, dry-run) being the transferable part.

## 6. Multi-tenant constraints single-user implementations never face

1. **Consent & spend:** every consolidation design costs extra LLM calls running unattended. In llame that's **BYOK spend against the tenant's own credentials** for a quality gain §3 says is unestablished. A background job silently spending a user's provider credits is a consent problem we would own. (OpenClaw's answer — the authoritative path costs zero LLM tokens — is the only shape that dodges this entirely.)
2. **Poisoning blast radius:** the failure-mode literature (SSGM [arXiv:2603.11768], AgentPoison lineage) converges on consolidation as _where_ injected/hallucinated content gets laundered into trusted long-term state. A consolidation worker in llame is a new privileged write path across tenants' data — it inherits the full "identity from trusted source, fail closed" scrutiny of any write surface, which no single-user reference has ever had to design for.
3. **Visibility/editability:** SPEC §20 requires curated memory be user-visible and editable. No sleep/dream stage in this research produces a **reviewable diff before commit**; they all silently update state. If llame ever builds consolidation, "show the user what changed, let them reject it" is a product requirement the literature hasn't solved (SSGM's reversible-reconciliation shape — mutable view + append-only log + rollback — is the closest proposal, unshipped).

## 7. Verdict for llame

1. **Episodic track (#194): no dreaming, full stop.** Hybrid search over verbatim history is the empirically winning configuration (§3); adding a consolidation stage would spend compute and add attack surface to move _away_ from it. (Cross-checked: this also matches OpenClaw's own recall architecture — its "dreaming" never touches how episodic content is retrieved.)
2. **Curated track (SPEC §20, when built): write-time, user-visible, no batch job.** Keep the extraction Mem0-shaped (write-time, scan-on-write per §20.3) and user-approved. Do not build an offline rewriter; the only argument for batching is cost (RecMem), and we have no measured cost problem.
3. **The dreaming-shaped ideas worth keeping, all deterministic:**
   - **Zero-LLM promotion ranker** over recall-event telemetry, recast as _candidate suggestion_ — "you keep looking this up; save it to memory?" — surfacing §20 write-policy prompts from usage instead of classifier guesses. Tenant-scoped `memory_recall_events` table (RLS FORCE), scheduled pg-boss job, AND-gate thresholds re-validated at apply time (OpenClaw's template).
   - **Citation-time recall logging** (the agent actually used the passage in its answer) as a deliberately stronger signal than OpenClaw's surface-time impressions — a design choice, cheap to make from day one. Forward-compat hook: `search_conversations` (#198) can emit recall-telemetry events — a table and an insert, committing to nothing.
   - **Bitemporal columns** (event-time vs ingestion-time, Zep/Graphiti) for staleness — a schema/retrieval decision that solves the "went to Singapore → was in Singapore" problem without any rewrite pass; consistent with the signal-columns plan already in the 2026-07-05 cross-report.
   - **Cheap-model triage gate + verdict cache + dry-run** (gbrain) as the cost-control shape _if_ an LLM distillation stage is ever justified later.
4. **Do not adopt:** reflection-tree synthesis, NREM/REM batch rewriting, idle-time context rewriters — no evidence base at our scale, and they sit exactly on the poisoning/drift/consent fault lines of §6.
