The first layer should be an **immutable event log**: every message, retrieval hit, tool result, edit, explicit memory save, and artifact reference is stored append-only. The second layer should be **short-term working memory** for the current run or chat band: rolling summary, active goal, temporary facts, open tool state, current plan. ## Governance, safety, and future-proofing for decay

The biggest reason not to treat memory as a simple retrieval add-on is that **persistent memory is a control surface and an attack surface**. PersistBench found high failure rates for cross-domain leakage and memory-induced sycophancy across many frontier and open models. The MemGate work argues that memory search itself is a trust boundary, because semantic similarity alone often retrieves memories that are related in embedding space but inappropriate in task context. Hidden in Memory then shows a more severe failure mode: persistent storage can be poisoned so that fabricated user facts are later retrieved and can drive downstream actions. ŅłĆciteŅłéturn15academia2Ņłéturn5academia3Ņłéturn4academia1Ņłü

That means the write path should be conservative. Do **not** auto-save everything that looks salient. Separate explicit user instructions such as ŌĆ£remember this,ŌĆØ high-confidence factual extraction, inferred profile traits, sensitive traits, and third-party information. Keep provenance on every memory item: source message, source document, timestamp, scope, extraction confidence, whether the memory is user-declared or model-inferred, and what later evidence reaffirmed or contradicted it. Consumer-memory analysis shows this matters: a study of real ChatGPT memory entries found that the majority were created unilaterally by the system, with a substantial amount involving personal or psychological information. That is acceptable for a consumer product only if users can inspect, edit, and delete it; for your architecture, it argues strongly for visible and reviewable memory management. ŅłĆciteŅłéturn8academia6Ņłéturn0file0Ņłéturn0file1Ņłü

This is also where your future ŌĆ£memory decayŌĆØ idea becomes relevant, even if you do not implement it yet. The mistake would be to hard-code decay as a single universal score too early. A better framework is to make the schema decay-ready by storing the signals that future policies can use: event time, last-confirmed time, source trust, extraction confidence, explicit user pinning, contradiction edges, supersession edges, read frequency, scope, and retention class. Then later you can experiment with retrieval-time weighting, consolidation-time promotion/demotion, or policy-learned forgetting without redesigning the substrate. Recent work on forgetting-aware evaluation and maintainable memory strongly supports this ŌĆ£keep enough structure now, optimise the forgetting policy laterŌĆØ approach. One further recommendation is to distinguish **forgetting** from **deletion**. For a serious assistant, the raw event log should usually remain immutable under retention policy, while higher-level semantic memory is what decays, gets superseded, or stops surfacing. That preserves auditability and lets you re-index later with a better policy. In other words, ŌĆ£memory decayŌĆØ should mostly mean ŌĆ£reduced eligibility for retrieval or promotion,ŌĆØ not ŌĆ£erase history from the only source of truth.ŌĆØ That is especially important in shared environments, where private vaults, shared memory, and assistant operations need a clear forensic trail.
